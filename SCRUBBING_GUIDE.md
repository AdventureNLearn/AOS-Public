# AOS-Public Scrubbing Guide

**Purpose**: This document defines the rules for maintaining operational security (OpSec) when contributing to or expanding the public AOS-Public repository.

## Core Principles

1. **No PII** — Never include real names, locations, specific projects, or personal identifiers.
2. **No Internal Paths** — Never reference internal Grok directories, file paths, or private tooling.
3. **No Third-Party Specifics** — Abstract or remove references to specific external organizations, tools, or individuals unless they are already public knowledge in a neutral way.
4. **Generalize Domain Work** — Specific civic, regulatory, or jurisdiction work should be abstracted into general principles and frameworks.
5. **Preserve Mission & Philosophy** — The Mission Spine (Christ Is King → America First → Truth-Seeking) and core values should remain visible where appropriate.

## What Belongs in This Repo

- Core orchestration and reasoning frameworks
- Replicable protocols and playbooks
- Activation fragments and prompt patterns
- High-level architecture and philosophy

## What Does NOT Belong

- Specific audits or investigations
- Personal projects or business operations
- Florida-specific or location-tied work (unless heavily abstracted)
- Internal conversation history or working documents
- Raw outputs from tools or agents

## Contribution Process

1. Create content in a private/internal environment first.
2. Run it through this Scrubbing Guide.
3. Submit only the cleaned version.
4. When in doubt, remove or generalize.

## Versioning Note

All public releases should be clearly versioned. Internal development can continue in the private environment.

*This guide exists to protect both the operator and the integrity of the public frameworks.*
